## 第 2 部分：商業目標 (Business Objectives) - 「為何做？」

*在投入資源前，我們必須清晰地定義問題、目標使用者和商業價值，避免開發出無人需要的產品。*

| 區塊 | 內容 |
| :--- | :--- |
| **1. 背景與痛點** | 目前台達內部技術知識（如設計規範、測試報告、Best Practice）雖已上傳至 TMF 平台，但存在嚴重的「上傳品質不一」與「知識孤島」問題。根據內部調查，超過 40% 的技術文件未正確標註應用領域或關鍵字，導致搜尋命中率低於 35%。同時，知識上傳流程依賴人工填寫問卷與手動分類，平均耗時 1.5 小時/篇，且缺乏自動化審核機制，造成知識更新滯後與重複建置。此問題嚴重影響研發效率，增加新專案的學習曲線與錯誤率 [[1]]。 |
| **2. 策略契合度** | 本專案直接支持台達「2025 數位轉型戰略」中的「知識資產化」與「AI 驅動決策」兩大支柱。透過自動化知識治理，實現「知識即資產」的願景，並為未來整合 AI 智能搜尋與建議引擎（如 Chatbot 類型的知識輔助系統）奠定資料基礎。同時，符合 DMS 推動全公司知識管理成熟度達 Level 4 的目標，提升跨 BU 的技術協作效率 [[11]]。 |
| **3. 成功指標 (Success Metrics)** | *我們如何衡量成功？定義 1-3 個可量化的關鍵績效指標 (KPI)。* <br> - **主要指標**: 上傳文件的「關鍵字標註準確率」提升至 90% 以上（現為 55%）<br> - **次要指標**: 技術文件平均搜尋響應時間降低至 1.5 秒內（現為 3.2 秒）<br> - **次要指標**: 每月有效上傳知識量提升 50%（現為 120 篇/月 → 目標 180 篇/月） |

## 第 3 部分：使用者故事與允收標準 (User Stories & UAT) - 「做什麼？」

*這是連接「商業需求」與「技術實現」的橋樑。每個使用者故事都應直接對應到下一個階段的 BDD 情境。*

### 核心史詩 (Core Epic): 知識上傳與自動分類

| 使用者故事 ID | 描述 (As a, I want to, so that) | 核心允收標準 (UAT) | 連結至 BDD 文件 |
| :--- | :--- | :--- | :--- |
| **US-001** | **As a** 技術工程師,<br>**I want to** 在上傳技術文件時，系統能根據文件內容自動建議「應用領域」與「關鍵字」,<br>**so that** 我能快速完成上傳，減少手動標註時間。 | 1. 上傳 PDF/DOCX 檔案後，系統於 10 秒內顯示 3 個最可能的應用領域（如 Firmware, Mechanical_Motor）<br>2. 系統自動提取 5 個高頻技術詞彙作為關鍵字候選<br>3. 使用者可修改或確認建議，確認後儲存 | [Link to `epic_knowledge_tagging.feature`] |
| **US-002** | **As a** 知識管理員,<br>**I want to** 系統能自動檢查上傳文件是否符合「品質評分標準」（如摘要完整性、機密等級正確性）,<br>**so that** 我能減少人工審核負擔，提升知識品質。 | 1. 系統在上傳時自動執行 5 項品質檢查（摘要長度>100字、關鍵字≥3個、機密等級非「公開」時有主管勾選）<br>2. 不符合項目以紅色警示標示，並提供修改建議<br>3. 未通過品質檢查的文件無法提交，需修正後重新上傳 | [Link to `epic_quality_check.feature`] |
| **US-003** | **As a** 專案經理,<br>**I want to** 查看「知識熱點地圖」，了解哪些技術領域的知識最活躍與最缺乏,<br>**so that** 我能優先推動知識補強計畫。 | 1. 系統提供互動式熱力圖，顯示各技術模組（如 EMI, Firmware）的上傳數量與平均評分<br>2. 標示「知識缺口」領域（上傳量<5篇且評分<3分）<br>3. 點擊領域可下鑽查看缺失文件類型建議 | [Link to `epic_knowledge_heatmap.feature`] |

## 第 4 部分：範圍與限制 (Scope & Constraints)

*明確定義邊界是專案成功的關鍵。這為 LLM 生成程式碼提供了最重要的「護欄」。*

| 區塊 | 內容 |
| :--- | :--- |
| **功能性需求 (In Scope)** | *高層次功能列表。*<br> - 自動內容分析與關鍵字建議（基於 NLP 模型）<br> - 知識品質自動審核引擎（規則引擎 + 模板比對）<br> - 知識熱點地圖與缺口分析儀表板<br> - 與現有 TMF 平台整合（單一登入、權限同步）<br> - 上傳流程中內嵌「問卷自動填補」功能（從文件內容提取背景、痛點、效益） |
| **非功能性需求 (NFRs)** | *系統必須滿足的品質屬性。*<br> - **性能**: 文件分析與建議響應時間 ≤ 10 秒（10MB 以內）<br> - **安全性**: 所有文件上傳與處理符合台達資訊安全等級 3（機密級）標準<br> - **可用性**: 支援 Chrome / Edge 瀏覽器，符合 WCAG 2.1 AA 標準，支援螢幕閱讀器<br> - **可擴展性**: 架構需支援未來新增 50+ 技術領域的分類
